# P1.2 — Ingen validering av AMU-sammensetning (AML §7-3)

## Prioritet: P1 (Høy daglig påvirkning + juridisk krav)

## Problem

`AmuParticipantsTable.tsx` viser deltakere med roller, men varsler ikke brukeren
dersom sammensetningen er i strid med loven.

**AML §7-3** krever:
1. Like mange representanter fra arbeidsgiver- og arbeidstakersiden.
2. Minst ett verneombud blant arbeidstakerrepresentantene (AML §7-4).
3. BHT-representant ved behov (AML §3-5).

I dag kan et møte signeres selv om det kun har arbeidsgiverrepresentanter og
ingen verneombud. Dette gjør referatet formelt ugyldig.

### Referansefiler (les disse FØR du redigerer noe)

- `modules/amu/AmuParticipantsTable.tsx` — hele filen (211 linjer)
- `modules/amu/AmuDetailView.tsx` — linjene 419–430 (preflight-sjekkliste),
  580–628 (planleggingsfanen som gjengir `AmuParticipantsTable`)
- `modules/amu/types.ts` — `AmuParticipantRole` (linjene ~14–21)
- `modules/amu/useAmu.ts` — `signMeetingAsChair` (linjene 549–583)

## Acceptance criteria

- [ ] En `WarningBox` vises øverst i deltakertabellen hvis antall
  arbeidsgiverrepresentanter (`employer_rep`) ≠ antall
  arbeidstakerrepresentanter (`employee_rep` + `safety_deputy`).
- [ ] En separat `WarningBox` vises hvis ingen deltaker har rolle
  `safety_deputy`.
- [ ] Advarslene er informative og hindrer ikke brukeren fra å lagre.
- [ ] Preflight-sjekklisten i signeringsfanen (`AmuDetailView.tsx` linjene
  425–430) inkluderer et nytt punkt: "Lik representasjon arbeidsgiver/ansatt".
- [ ] Preflight-punktet er grønt (ok) kun når antallet er balansert OG minst
  ett verneombud er med.
- [ ] Advarslene vises ikke for signerte møter.
- [ ] `npm run build` exits 0.

## Implementeringstrinn

### Trinn 1 — Beregningslogikk i `AmuParticipantsTable.tsx`

Les hele filen. Legg til to beregnede verdier rett før `return`-setningen (etter
linje 67):

```tsx
const employerCount = participants.filter(
  (p) => p.role === 'employer_rep',
).length

const employeeCount = participants.filter(
  (p) => p.role === 'employee_rep' || p.role === 'safety_deputy',
).length

const hasSafetyDeputy = participants.some((p) => p.role === 'safety_deputy')

const isBalanced = employerCount > 0 && employeeCount > 0
  && employerCount === employeeCount
```

### Trinn 2 — Vis advarsler i `AmuParticipantsTable.tsx`

Legg til import av `WarningBox` øverst i filen:

```tsx
import { WarningBox } from '../../src/components/ui/AlertBox'
```

I `return`-blokken, legg til advarslene rett etter åpnende `<>` og FØR
`<ModuleRecordsTableShell>` (etter linje 83):

```tsx
{!readOnly && participants.length > 0 && !isBalanced && (
  <WarningBox>
    <strong>AML §7-3:</strong> AMU skal ha lik representasjon fra begge sider.
    Arbeidsgiver: {employerCount}, Arbeidstaker (inkl. verneombud): {employeeCount}.

  </WarningBox>

{!readOnly && participants.length > 0 && !hasSafetyDeputy && (
  <WarningBox>
    <strong>AML §7-4:</strong> Minst ett verneombud skal delta i AMU-møtet.
    Ingen deltaker har rolle «Verneombud».
  </WarningBox>

```

### Trinn 3 — Eksporter balanse-status som prop (valgfri tilnærming)

I stedet for å duplisere logikken i `AmuDetailView.tsx`, eksporter en ny
`participantBalanceOk`-verdi fra `AmuParticipantsTable`-propsene:

Legg til en ny callback-prop i `AmuParticipantsTableProps`
(`AmuParticipantsTable.tsx` linjene 22–32):

```tsx
onBalanceChange?: (balanceOk: boolean) => void
```

Legg til en `useEffect` i komponenten som kaller `onBalanceChange` når
`isBalanced` eller `hasSafetyDeputy` endrer seg:

```tsx
useEffect(() => {
  onBalanceChange?.(isBalanced && hasSafetyDeputy)
}, [isBalanced, hasSafetyDeputy, onBalanceChange])
```

## Anti-patterns

- **Ikke** blokker signeringen på grunn av ubalansert sammensetning — AML
  tillater at møter holdes med avvikende sammensetning ved særlige
  omstendigheter. Advarsler er nok.
- **Ikke** vis advarslene for signerte møter (`readOnly = true`) — de er
  allerede frosset.
- **Ikke** tell `bht` og `secretary` som verken arbeidsgiver- eller
  arbeidstakerside — de er observatører.
