# P1.3 — Agendasortering er ikke-intuitiv (manuell rekkefølge-nummer)

## Prioritet: P1 (Høy daglig påvirkning)

## Problem

Agendapanelet i `AmuDetailView.tsx` (linjene 790–799) bruker et numerisk
inputfelt for `order_index` (rekkefølge 0 = først). Brukeren ser "3" i
sakliste-tabellen (fordi `a.order_index + 1` vises) men må skrive "2" i feltet
for å oppnå samme posisjon. Dette er forvirrende.

I tillegg er det ingenting som forhindrer to saker med samme `order_index`, noe
som gir udefinert sortering.

Sammenlign: SafetyCulture, KaiNexus og Notion har alle dra-og-slipp-sortering
for listeelementer i møter.

Løsningen for denne sprinten er et forenklet alternativ til dra-og-slipp:
"Flytt opp / Flytt ned"-knapper ved siden av hvert agendapunkt. Dette er en
`O(n)`-operasjon på klientside med én batch-oppdatering mot Supabase.

### Referansefiler (les disse FØR du redigerer noe)

- `modules/amu/AmuDetailView.tsx` — linjene 790–799 (order_index-felt),
  280–320 (`saveAgendaFromPanel`), 302–316 (agendaoppdatering)
- `modules/amu/AmuAgendaPlanningTable.tsx` — hele filen (147 linjer)
- `modules/amu/useAmu.ts` — linjene 675–712 (`updateAgendaItem`)
- `modules/amu/types.ts` — `AmuAgendaItem`

## Acceptance criteria

- [ ] Tabellen i `AmuAgendaPlanningTable` har pil-opp og pil-ned-knapper for
  hvert agendapunkt.
- [ ] Klikk på pil-opp bytter `order_index` for gjeldende sak med saken over.
- [ ] Klikk på pil-ned bytter `order_index` for gjeldende sak med saken under.
- [ ] Etter rekkefølgebytte sendes `updateAgendaItem` for begge berørte saker,
  og tabellen re-sorterer umiddelbart i UI.
- [ ] Pil-opp er deaktivert for første sak; pil-ned for siste sak.
- [ ] Det manuelle `order_index`-nummerfelt i panelet fjernes.
- [ ] Tabellen viser alltid 1-basert rekkenummer (1, 2, 3…) — aldri 0-basert.
- [ ] Knappene er deaktivert for signerte møter (`readOnly = true`).
- [ ] `npm run build` exits 0.

### Trinn 3 — `onReorder`-handler i `AmuDetailView.tsx`

Les linjene 280–330 i `AmuDetailView.tsx`. Legg til ny callback etter
`deleteAgenda` (etter linje 331):

```tsx
const reorderAgenda = useCallback(
  async (itemId: string, direction: 'up' | 'down') => {
    const sorted = [...agenda].sort((a, b) => a.order_index - b.order_index)
    const idx = sorted.findIndex((a) => a.id === itemId)
    if (idx < 0) return
    const swapIdx = direction === 'up' ? idx - 1 : idx + 1
    if (swapIdx < 0 || swapIdx >= sorted.length) return

    const current = sorted[idx]
    const swap = sorted[swapIdx]
    const currentNewIndex = swap.order_index
    const swapNewIndex = current.order_index

    // Oppdater begge på én gang (parallell)
    const [updCurrent, updSwap] = await Promise.all([
      amu.updateAgendaItem(current.id, { order_index: currentNewIndex }),
      amu.updateAgendaItem(swap.id, { order_index: swapNewIndex }),
    ])

    if (updCurrent && updSwap) {
      setAgenda((prev) =>
        prev
          .map((a) => {
            if (a.id === updCurrent.id) return updCurrent
            if (a.id === updSwap.id) return updSwap
            return a
          })
          .sort((a, b) => a.order_index - b.order_index),
      )
    }
  },
  [agenda, amu],
)
```

### Trinn 5 — Fjern `order_index`-felt fra agendapanelet

I agendapanel-skjemaet i `AmuDetailView.tsx` (linjene 790–799), slett hele
`WPSTD_FORM_ROW_GRID`-blokken for "Rekkefølge (0 = først)".

Oppdater `openNewAgendaPanel` (linjene 258–267) for å beregne
`order_index` automatisk:

```tsx
const openNewAgendaPanel = useCallback(() => {
  const maxIndex = agenda.length === 0
    ? 0
    : Math.max(...agenda.map((a) => a.order_index)) + 1
  setAgendaForm({
    title: '',
    description: '',
    order_index: maxIndex,
    source_module: '',
    source_id_raw: '',
  })
  setAgendaPanel({ mode: 'new' })
}, [agenda])
```

## Anti-patterns

- **Ikke** implementer dra-og-slipp med ekstern lib (f.eks. `dnd-kit`) — det er
  ikke i `package.json` og legger til unødvendig kompleksitet for denne sprint.
- **Ikke** re-nummerer alle `order_index`-verdier ved hvert bytte — bytt bare
  de to berørte sakene.
- **Ikke** blokker UI mens begge oppdateringer kjører — vis den optimistiske
  sorteringen umiddelbart, rull tilbake kun ved feil.
